# 并行求解工作流详细指南

## 一、头脑风暴方法论

### 差异化方案生成策略

生成方案时应沿以下维度进行差异化思考，确保每个方案走的是**不同的路**：

#### 1. 技术路线差异

- 使用不同的库/框架实现同一功能
- 原生实现 vs 第三方库实现
- 同步方案 vs 异步方案
- 面向对象 vs 函数式 vs 过程式

#### 2. 架构层级差异

- 修改底层实现 vs 修改上层调用
- 改源码 vs 加中间件/拦截器/代理
- 修改数据结构 vs 修改算法逻辑
- 局部修复 vs 全局重构

#### 3. 策略方向差异

- 正面解决问题 vs 绕过问题（workaround）
- 修改现有代码 vs 替换为全新实现
- 增量修复 vs 回退到已知可行状态再重做
- 精确修复 vs 容错/降级策略

#### 4. 问题定位差异

- 如果根因不确定，可以基于不同的根因假设分别制定方案
- 前端问题 vs 后端问题 vs 配置问题 vs 环境问题

### 方案数量决策参考

| 场景 | 建议方案数 |
|------|-----------|
| 根因明确，需要尝试不同修复路线 | 2-3 个 |
| 根因不确定，需要多方向探索 | 3-5 个 |
| 问题复杂，涉及多个子系统 | 4-6 个 |
| 极端情况，需要全面覆盖 | 7-10 个 |

**注意**：方案数量不是越多越好，每个方案都应该有合理的成功预期。避免为了凑数而加入明显不可行的方案。

### 方案描述模板

```markdown
## 方案 N：[方案名称]

**核心思路**：[详细说明]

**技术路线**：
- [使用什么技术/库/模式]
- [与其他方案的关键区别]

**实现步骤**：
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

**验收标准**：
- [标准 1]
- [标准 2]

**风险评估**：[可能的风险点]
```

## 二、任务文件生成规范

### 任务描述内容要求

每个任务描述中必须包含足够的上下文，因为每个 worktree 中的 Claude Code 是**独立运行**的，它看不到当前对话的上下文。因此：

1. **完整的问题描述**：不能假设 agent 知道问题是什么
2. **已失败方案的信息**：告知哪些路已经走不通，避免重蹈覆辙
3. **具体的文件路径**：明确需要修改的文件位置
4. **预期的行为**：清晰定义什么是"修好了"
5. **约束条件**：不能破坏的现有功能、不能使用的方案等

### 注意事项区域内容

注意事项应包含：
- 项目的技术栈和关键依赖版本
- 不能影响的现有功能
- 代码规范要求
- 已知的坑和限制

## 三、验证判断逻辑

### 是否需要 `-r` 运行指令的判断流程

```
项目是否使用热更新/热重载？
├── 是（Vite / Webpack Dev / Next.js Dev / CRA 等）
│   └── 不需要 -r（代码变更可直接观察）
└── 否
    ├── 项目有测试文件？
    │   ├── 是 → 需要 -r，使用测试命令
    │   │       例：-r "npm run test" / -r "pnpm test" / -r "vitest run"
    │   └── 否 → 继续判断
    ├── 项目需要构建？
    │   ├── 是 → 需要 -r，使用构建命令
    │   │       例：-r "npm run build" / -r "pnpm build"
    │   └── 否 → 不需要 -r
    └── 两者都需要？
        └── 组合使用：-r "npm run test & npm run build"
```

### 验证结果评估标准

| 验证项 | 通过标准 |
|--------|---------|
| 代码应用 | `clawt validate` 命令执行成功，无 patch 冲突 |
| 运行指令 | `-r` 指定的命令执行返回 exit code 0 |
| 功能正确性 | 在验证分支上检查代码逻辑是否符合预期 |

### 人工检查场景

以下情况需要人工辅助确认验证结果（即使 `-r` 通过）：
- 性能优化（需要基准测试数据）
- 涉及外部服务交互的变更（需要联调验证）

### 验证闭环：每个方案必须自主验证

每个并发执行的独立方案都必须设计自己的验证方式，确保能自主判断"问题是否真正解决"。验证闭环的设计应写入任务描述中，让 agent 在实现方案时同步完成验证逻辑。

#### 前端项目：browser-tools MCP

通过 browser-tools MCP 工具构建验证闭环：

**可用的 MCP 工具：**

| 工具 | 用途 |
|------|------|
| `getConsoleLogs` | 获取浏览器控制台所有日志 |
| `getConsoleErrors` | 获取浏览器控制台错误日志 |
| `getNetworkErrors` | 获取网络请求错误 |
| `takeScreenshot` | 截取当前页面截图，进行视觉验证 |
| `runDebuggerMode` | 运行调试模式，深度分析问题 |

**验证闭环构建方式：**

1. **添加调试日志**：在每个方案的任务描述中，指导 agent 在关键位置添加 `console.log` 调试信息（如状态变化、函数执行结果、数据流向等）
2. **热更新生效**：前端项目代码修改后，热更新会自动生效，调试日志会立即输出到浏览器控制台
3. **读取日志验证**：通过 `getConsoleLogs` / `getConsoleErrors` 获取浏览器日志，检查是否有报错或预期的调试输出
4. **截图验证**：通过 `takeScreenshot` 截取页面截图，确认 UI 渲染结果是否符合预期
5. **网络请求验证**：通过 `getNetworkErrors` 检查是否有 API 请求失败

#### 后端 / CLI / 库项目

通过以下方式构建验证闭环：

- **编写针对性测试用例**：在任务描述中要求 agent 编写覆盖修复场景的单元测试或集成测试
- **执行命令并比对输出**：对于 CLI 工具，运行命令并检查输出是否符合预期
- **检查日志**：添加关键日志输出，执行后检查日志内容
- **编写验证脚本**：编写独立的验证脚本，自动化检查修复结果

## 四、循环修正策略

### 核心原则：不放弃任何独立方案

当所有方案都失败时，**全部继续修正**，不筛选、不放弃任何一个独立方案。每个方案走的是不同的技术路线，任何一个都可能在修正后成功。

### Resume 修正流程

1. **分析失败日志**：从 clawt 输出中提取每个方案的执行结果
2. **定位失败原因**：是代码错误、逻辑错误、还是方向性问题
3. **编写 resume 指令**：在原始任务文件中更新每个失败方案的任务描述，追加上一轮的失败信息和修正方向
4. **批量 resume**：使用 `clawt resume -f` 一次性并发 resume 所有方案

   ```bash
   clawt resume -f <原始任务文件路径>
   ```

### Resume 任务描述更新模板

在原始任务文件中，将需要 resume 的方案任务描述更新为：

```markdown
<!-- CLAWT-TASKS:START -->
# branch: clawt-solve-<方案名>
## 继续修正：[方案名称]

### 上一轮执行结果
[描述上一轮做了什么、结果如何]

### 失败原因分析
[具体的失败原因]

### 本轮修正方向
[需要做什么修改来解决问题]

### 验收标准
[与第一轮相同，但可以增加针对性检查项]
<!-- CLAWT-TASKS:END -->
```

### 循环上限控制

| 轮次 | 动作 |
|------|------|
| 第 1 轮 | 全量并行执行所有方案 |
| 第 2-5 轮 | 全部失败方案 resume 修正（不递减，不放弃任何方案） |
| 超过 5 轮 | 停止循环，汇总所有进展报告给用户 |

### 超过循环上限后的汇报内容

```
## 并行求解汇总报告

### 问题
[问题描述]

### 尝试的方案
1. 方案 A（clawt-solve-xxx）：[状态] - [简要说明]
2. 方案 B（clawt-solve-xxx）：[状态] - [简要说明]
...

### 最接近成功的方案
[哪个方案最有希望，目前到什么程度]

### 建议的下一步
[基于所有尝试结果，给出继续排查的建议]

### 相关分支
- clawt-solve-xxx：[可以手动检查的内容]
- clawt-solve-xxx：[可以手动检查的内容]
```

## 五、特殊场景处理

### clawt 未安装

如果执行 `clawt` 命令时报错（命令不存在），提示用户：
```
clawt 工具未安装。请先安装：npm install -g clawt
安装后需要在项目中初始化：clawt init
```

### 项目未初始化

如果 `clawt run` 报错提示未初始化，自动执行：
```bash
clawt init
```

### Patch 冲突

如果 `clawt validate` 时遇到 patch 冲突：
1. 先尝试 `clawt sync -b <分支名>` 同步主分支
2. 然后重新 `clawt validate`
3. 仍然失败则标记该方案为"需要手动解决冲突"

### 所有方案都方向性错误

如果分析后认为所有方案的方向都不对：
- 不进入 resume 循环
- 直接向用户报告分析结论
- 建议重新定义问题或提供更多上下文
