# 边界条件追根究底式覆盖分析

## 流程总览

```
步骤 1            步骤 2              步骤 3              步骤 4
收集上下文    →   追问分析      →    输出方案      →    执行修改
git diff         对每步用 7 问       全局清单表格          按低→中→高
构建链路         广度 + 深度         逐点方案 ABCD         分批验证
                 必要时读理论参考     分段等批准
```

**核心原则**：不再随机罗列边界条件，而是沿着本次改动的上下游链路按时间顺序逐层下钻。如需深入理解分析方法背后的理论依据，读 `references/theory-toolbox.md`。

---

## 步骤 1：收集上下文 & 构建链路

### 1.1 获取改动

```bash
git diff --cached --stat && git diff --stat   # 摘要
git diff --cached && git diff                  # 完整 diff
```

从 diff 中提取被修改的函数名、组件名、方法名。

### 1.2 阅读源码确认

对每个改动函数，确认三件事：
- **函数签名**：入参类型、返回值类型、可能抛什么异常
- **调用方**：谁在什么条件下调用它（用 LSP findReferences）
- **消费方**：返回值/副作用被谁依赖、异常会传播到哪

### 1.3 构建三段链路

以改动函数为圆心，向上下游各扩展一层：

```
[上游] 调用方传参 → [核心] 本次改动逻辑 → [下游] 返回值消费 / 异常传播
```

**示例**：改动 `validateFile(file)` 函数中 `>` 改为 `>=`

```
【上游 - 调用方】                    【核心 - validateFile】           【下游 - 消费方】
U1: handleDrop(e) 提取 files[0]  →   C1: if (!file) return error    D1: result.valid 决定是否加入列表
U2: handlePaste(e) 提取 files[0] →   C2: if (size >= MAX) reject ←  D2: FileList 根据 valid 渲染状态
U3: <FileInput onChange={...}>    →   C3: if (!allowed type) reject   D3: 若 throw → 无 catch → 白屏
                                      C4: return { valid: true }
```

**原则**：
- 默认只拆到上下游各一层，不展开全链路（除非发现传播链更长且关键）
- 上游关注参数来源的多样性（不同调用方传的值可能差异很大）
- 核心关注每个分支和状态转换的边界
- 下游关注返回值/异常的传播路径（null 传到哪里会炸？）

> 如果你对"为什么用契约视角分析上下游"、"如何精确列出边界值"、"风险优先级怎么量化"等方法论感兴趣，在分析过程中随时读 `references/theory-toolbox.md`。它包含：契约式设计、边界值分析/等价类划分、FMEA 三维评估、切斯特顿栅栏、混沌工程思维、5 Why 深挖、收益递减定律。

---

## 步骤 2：追问分析

对链路中的每个步骤（上游每一个调用方、核心每一个分支、下游每一个消费方），做两轮追问。

### 2.1 广度追问（7 个维度，必做）

1. **如果这一步失败了，会怎样？**（主动和被动失败都要考虑）
2. **如果用户在此时做一个异常操作，会怎样？**（快速点击、切换标签、刷新、杀进程）
3. **如果输入数据是极端值，会怎样？**（0、空、负数、超大、特殊字符、二进制）
4. **如果并发执行两次，会怎样？**（重复提交、幂等性）
5. **如果依赖的外部系统异常，会怎样？**（网络、存储、权限、其他服务）
6. **这一步卡住了/超时了，用户能做什么？**（有取消按钮吗？有反馈吗？）
7. **这一步的状态在前端、后端、UI 三者间是否一致？**（数据源唯一性、同步机制）

### 2.2 补充检查维度

| 维度 | 核心问题 |
|------|---------|
| **输入边界** | 空值、极端值、非法值、特殊字符、超长/超大、类型不符 |
| **时序竞态** | 快速连续操作、并发请求、操作未完成又触发了新操作 |
| **状态冲突** | 组件生命周期内状态不一致、多个组件同时操作同一数据 |
| **环境异常** | 网络中断/切换、浏览器兼容、内存不足、存储满 |
| **用户中断** | 中途取消、关闭页面、切换标签、浏览器崩溃恢复 |
| **并发/重复** | 幂等性、去重、重复提交、重试与首次的关系 |
| **UI 边界** | 超长文本截断、大量数据渲染、loading/empty/error 三态 |
| **资源清理** | 内存泄漏、事件监听未解绑、定时器未清除 |

### 2.3 深度下钻（5 Why，按需触发）

当某个边界点的表面修复太简单（"加个 if 判空就行了"），或同一类问题反复出现时，切换到 5 Why 模式追问架构根因。

### 2.4 边界判定

每个步骤追问完后给出明确结论：
- **列入清单**：当前无防护且值得修复
- **不适用**：此风险对此步骤不存在（如同步函数不存在"异步中断"问题，直接跳过）
- **有意设计**：查 git blame 确认是作者故意省略（切斯特顿栅栏），标注跳过
- **不值得修**：低发生度 + 低严重度，收益递减定律建议不修

## 步骤 3：输出边界清单与方案

### 3.1 全局清单表格

所有边界点合并为一个表格：

| # | 步骤 | 边界点 | S | O | D | RPN | 状态 | 复杂度 |
|---|------|--------|---|---|---|---|---------|------|
| 1 | C2 | 空文件 size=0 被正常上传 | 3 | 3 | 5 | 45 | ❌ | 低 |
| 2 | C2 | MAX_SIZE 常量无编译时拦截 | 2 | 1 | 2 | 4 | ❌ | 低 |
| 3 | D3 | 调用方无 try-catch 致白屏 | 4 | 2 | 4 | 32 | ❌ | 中 |

**S/O/D 评分标准（1-5）**：

| 分 | 严重度 S | 发生度 O | 检测度 D |
|----|---------|---------|---------|
| 1 | 用户无感知 | 需刻意构造 | 编译时/类型系统即报错 |
| 2 | UI 异常不影响功能 | 特定环境/条件 | 单测必然失败 |
| 3 | 功能受阻但可绕过 | 非正常操作路径 | CR 有经验能发现 |
| 4 | 核心流程中断 | 正常路径偶尔触发 | 线上监控能发现 |
| 5 | 白屏/数据损坏/安全漏洞 | 每次必现 | 用户投诉才知道 |

**RPN 分级**：≥60 阻塞上线 | 30-59 本迭代 | 15-29 下迭代 | <15 记录不修或顺手改

### 3.2 逐点详细分析（按复杂度分段，每段暂停等批准）

**复杂度定义**：
- **低**：简单判空/try-catch/防抖，每个 < 30 行
- **中**：状态管理重构/前后端协同校验，每个 30-100 行
- **高**：架构调整/新服务/数据迁移，每个 > 100 行

**单个边界点模板**：

```
## 边界点 #N：[步骤] [名称]

当前行为：[代码实际行为]
预期行为：[理想行为]
FMEA：S=X O=X D=X → RPN=XX
触发条件：[操作路径]

### 方案 A：[描述]
思路：[3-5 句]
优点：- ... 缺点：- ...
改动量：X 文件 / ~X 行

### 方案 B：[描述]
...

### 🏆 推荐：方案 X
理由：[1-2 句]
```

**一段结束后必须输出**：

> 📋 **低复杂度边界点共 N 个。请审核每个推荐方案，确认后回复"低复杂度批准"或逐点反馈。**

**必须等待批准后才进入下一复杂度段。全部三段批准后才进入步骤 4。**

---

## 步骤 4：执行修改

按低 → 中 → 高顺序修改。每完成一批，告知用户可手动验证，再继续下一批。

优先修改 RPN 高的边界点，同复杂度内按 RPN 降序处理。

---

## 完整示例

> 任务："修复 validateFile 大小校验 bug：file.size > MAX_SIZE 改为 >="

**步骤 1**：`git diff --cached` → 改动 `validateFile` 一行。LSP 找引用 → 3 个上游调用方，2 个下游消费路径。构建三段链路（见上文 1.3 示例）。

**步骤 2**：对 C1（判空）、C2（大小比较）、D3（异常传播）各走 7 问。

- C1：已有判空，假 File 对象（mock）低概率低后果，→ 不列入
- C2：边界值修复已正确 ✅。追问发现 0 字节文件被放行 → 列入。追问发现 MAX_SIZE 常量可被误改 → 列入。
- D3：caller 无 try-catch，内部 throw 则白屏 → 列入。

**步骤 3**：输出全局表格（见上文 3.1），然后按复杂度分段。

## 边界点 #1：C2 - 空文件 size=0 应拒绝

当前行为：0 字节通过校验。预期：拒绝并提示。
FMEA：S=3 O=3 D=5 → RPN=45

方案 A：加 `if (file.size === 0) return error`。1 文件 / ~3 行。
方案 B：返回 `{ valid: true, warning }`。3 文件 / ~15 行。
🏆 推荐 A：空文件 99% 是误操作，直接拒绝最简单。

## 边界点 #2：C2 - MAX_SIZE 常量无校验

当前行为：常量可被误改为负数。预期：单元测试守护。
FMEA：S=2 O=1 D=2 → RPN=4

方案 A：单测验证常量范围。1 文件 / ~10 行。
方案 B：运行时校验。1 文件 / ~5 行。
🏆 推荐 A：配置错误是开发期 bug，单测是最低成本拦截点。

> 📋 **低复杂度边界点共 2 个。请审核，确认后回复"低复杂度批准"或逐点反馈。**

*[等待...]*

## 边界点 #3：D3 - 调用方无 try-catch

当前行为：exception → 白屏。预期：safe wrapper 捕获。
FMEA：S=4 O=2 D=4 → RPN=32

方案 A：三个 caller 各处加 try-catch。1 文件 / ~30 行。
方案 B：封装 `safeValidateFile()` 统一捕获。2 文件 / ~20 行。
🏆 推荐 B：集中防护优于分散重复，"永抛不异常"是强契约。

> 📋 **中复杂度边界点共 1 个。请审核...**

*[全部批准后进入步骤 4]*

**步骤 4**：先改 #1（加判空 3 行），再改 #2（写单测 10 行），最后改 #3（封装 safe wrapper）。每步完成提示验证。

## 执行准则

1. **范围默认 ±1 层**：上游到调用方，下游到消费方，不蔓延全链路。
2. **先看代码再分析**：必须读源码确认行为，不靠猜测。
3. **不跳过追问**：每步走完 7 问才做边界判定。
4. **果断跳过**：追问后确认不存在风险，直接标"不适用"，不硬凑。
5. **不跨段推进**：一段未批准不进下一段。
6. **不替用户做决定**：给 2-3 个方案 + 推荐理由，用户拍板。
7. **知道何时停**：RPN < 15 的边界可直接标"不修复"。
8. **方法参考**：分析卡住时，读 `references/theory-toolbox.md` 找灵感——它告诉你怎么契约分析、怎么列精确边界值、怎么五问挖根因。
